## ADR-006: Валюта на платформе

**Дата:** 03.05.2026  
**Статус:** PROPOSED — требуется решение Тимура  
**Контекст:** В модели `Load` два поля — `price_usd` и `price_gel`. В модели `Response` только `price_usd`. При принятии отклика сделка может получить цену в разных валютах — груз в ₾, отклик в $. В `Deal.currency` поле есть, но `agreed_price` берётся из отклика (всегда USD) или груза (может быть GEL). Фронт отображает `₾` или `$` в зависимости от того, какое поле заполнено.

**Проблема:** В одной сделке возможен микс валют. Экспорт для rs.ge некорректен, если стороны согласовали цену в разных валютах.

---

### Вариант A — Всё в лари (GEL), USD только для справки

**Суть:** `Load.price_gel` — основное поле, `price_usd` и `Response.price_usd` переименовать в `price_gel`. Хранить только GEL. Таблицы ставок ($/км) — только для отображения, не хранить в БД.

**Плюсы:**
- Грузия — страна с лари. Для местных сделок GEL логичнее.
- Простая логика: одна валюта, нет конвертации.
- rs.ge экспорт → всегда GEL, без вопросов.
- Минимум изменений в UI.

**Минусы:**
- Международные маршруты (Россия, Турция) — перевозчики привыкли считать в USD/RUB.
- Нет возможности зафиксировать долларовую договорённость.
- При инфляции лари — цены надо пересчитывать вручную.

**Что мигрировать:**
- `Response.price_usd` (Integer!) → `price_gel` (Float)
- Все записи в таблице `responses` — конвертировать по курсу на дату создания (или принять as-is, если немного данных)
- Убрать `Load.price_usd` из API-ответов или оставить для отображения
- Фронт: убрать `cur: "$"` ветку, всегда показывать `₾`

**Время реализации:** ~2-3 часа

---

### Вариант B — Всё в USD, лари только отображение

**Суть:** Единственная база-валюта — USD. На фронте конвертировать в GEL по курсу (из внешнего API или захардкоженному).

**Плюсы:**
- Стабильнее для межнациональных маршрутов.
- Перевозчики из РФ/Турции привыкли к USD.

**Минусы:**
- rs.ge требует GEL для отчётности. Нужна конвертация в момент экспорта.
- Нужен курс USD/GEL (Нацбанк Грузии API или захардкодить — оба варианта плохие).
- Юридически договоры в Грузии лучше в GEL.
- Клиенты-грузины видят цену в $ — непривычно.

**Что мигрировать:**
- `Load.price_gel` → конвертировать в USD и сохранить в `price_usd`
- Добавить `exchange_rate` snapshot в `Deal` (курс на момент сделки)
- Подключить API курсов (NBG API бесплатный: nbg.gov.ge)

**Время реализации:** ~1 день + интеграция с API курсов

---

### Вариант C — Две валюты + снапшот курса в момент сделки (рекомендован для будущего)

**Суть:** Хранить цену в валюте которую выбрал пользователь + добавить в `Deal` поля `price_gel_snapshot` и `exchange_rate_snapshot`. При создании сделки — записать курс на этот момент.

**Плюсы:**
- Максимальная гибкость: договорились в $, зафиксировали в ₾ для rs.ge.
- Исторически точно: курс зафиксирован на дату.
- Никто ничего не теряет при колебании курса.

**Минусы:**
- Сложнее реализация.
- Нужен курс NBG в реальном времени.
- Два поля цены везде — UX усложняется.

**Что мигрировать:**
- Добавить в `Deal`: `price_gel_snapshot Float`, `price_usd_snapshot Float`, `exchange_rate Float`
- Добавить в `Response`: поле `currency String` (GEL/USD)
- При `accept_response` → запросить NBG API и записать снапшот

**Время реализации:** ~1-2 дня

---

### Рекомендация (не решение)

Для **быстрого запуска** — Вариант A (всё в GEL): минимум изменений, Грузия = GEL.
Для **масштабирования** — Вариант C через 3-6 месяцев после первых международных сделок.

**Решение принимает:** Тимур  
**Дедлайн решения:** до старта рекламных кампаний

## ADR-007: Справочник городов

**Дата:** 03.05.2026  
**Статус:** PROPOSED — требуется решение Тимура  
**Контекст:** `Load.from_city`, `Load.to_city`, `Truck.available_from`, `Truck.available_to` — все хранятся как свободный текст. В БД уже могут быть записи «Тбилиси», «тбилиси», «Тбилилсси», «Tbilisi». Фильтр `ilike` частично спасает, но: агрегация по городам сломана, автокомплит нет, опечатки попадают в БД.

**Проблема:** Нет нормализации → фильтры неточные, статистика врёт, поиск «из Тбилиси» не находит «tbilisi».

---

### Вариант A — Захардкодить топ-50 городов (JSON-справочник)

**Суть:** Создать файл `app/data/cities.json` с топ-50 городами (Грузия, Россия/СНГ топ-20, Турция топ-5). На фронте — datalist или select. API валидирует против этого списка.

```json
[
  {"id": "tbilisi", "label_ru": "Тбилиси", "label_ge": "თბილისი", "country": "GE"},
  {"id": "batumi",  "label_ru": "Батуми",  "label_ge": "ბათუმი",  "country": "GE"},
  ...
]
```

**Плюсы:**
- Реализация за 2-3 часа.
- Нет внешних зависимостей и расходов.
- Фильтры работают точно.
- Покрывает 95%+ реальных маршрутов.

**Минусы:**
- Жёсткий список: нельзя добавить новый город без деплоя.
- Для нетипичных маршрутов — пользователь застрял.
- Нужно переиндексировать старые записи в БД (разовая задача).

**Стоимость:** $0. Время: 3-4 часа (JSON + валидация на бэке + datalist на фронте).

---

### Вариант B — Google Places API с автокомплитом

**Суть:** Поле города → текст с автокомплитом через Google Places API. При выборе — сохранять `place_id` + нормализованное название.

**Плюсы:**
- Любой город мира.
- UX: привычный автокомплит как в Google Maps.
- Нормализация гарантирована Google.

**Минусы:**
- **Стоимость:** $0.003/запрос. При 1000 пользователях и 5 запросах каждый — $15/мес минимум, растёт с трафиком.
- Нужен API ключ, биллинг аккаунт Google Cloud.
- При превышении лимита — форма ломается.
- GDPR: запросы идут через Google.
- Зависимость от внешнего сервиса.

**Стоимость:** $15-100/мес + разработка 1 день.

---

### Вариант C — Таблица в БД с админкой

**Суть:** Таблица `cities(id, name_ru, name_ge, country, is_active)`. Админ добавляет города через панель. Бэк валидирует поле города против этой таблицы.

**Плюсы:**
- Гибко: можно добавлять города без деплоя.
- Нет внешних зависимостей.
- Статистика по городам точная.

**Минусы:**
- Нужна админка (ещё не реализована).
- Время реализации: 1-2 дня (таблица + CRUD + валидация + миграция).
- До заполнения таблицы — фича не работает.

**Стоимость:** $0 + 1-2 дня разработки.

---

### Вариант D — Свободный текст, но нормализация (lowercase + trim)

**Суть:** Оставить свободный ввод. При сохранении: `city = city.strip().title()` (первая буква заглавная, остальные маленькие). Добавить `ilike` в фильтрах (уже есть).

**Плюсы:**
- Нет изменений в UX.
- Реализация за 30 минут.
- Покрывает 80% дублей (опечатки — нет, но капитализация — да).

**Минусы:**
- «Тбилиси» и «Tbilisi» — всё равно разные записи.
- Опечатки в БД остаются.
- Нет автокомплита — пользователь вводит руками.
- Агрегация по городам по-прежнему неточная.

**Стоимость:** $0. Время: 30 минут.

---

### Рекомендация (не решение)

**Быстрый старт:** Вариант D (30 мин) + Вариант A параллельно (3-4 часа).  
**Стратегически:** Вариант A — оптимален для текущего масштаба (Грузия + соседи).  
Вариант B — только если трафик вырастет до 10K+ MAU и есть рекламный бюджет.

**Решение принимает:** Тимур

## ADR-006: Валюта на платформе — ACCEPTED

**Дата:** 03.05.2026  
**Статус:** ACCEPTED  
**Решение:** Вариант C — две валюты со снапшотом курса NBG.

## ADR-007: Справочник городов — ACCEPTED

**Дата:** 03.05.2026  
**Статус:** ACCEPTED  
**Решение:** Yandex Geocoder API (Advanced) + своя таблица `cities` как fallback.

**Детали:**

**Миграция:**
```sql
CREATE TABLE cities (
    id SERIAL PRIMARY KEY,
    name_ru VARCHAR NOT NULL,
    name_ge VARCHAR,
    country_iso CHAR(2) NOT NULL,
    lat FLOAT,
    lon FLOAT,
    yandex_geo_id VARCHAR
);
```

## ADR-010: GDPR-удаление аккаунта

**Дата:** 03.05.2026
**Статус:** PROPOSED — ожидает решения Тимура
**Контекст:** Грузия — страна-ассоциат ЕС, GDPR применяется де-факто. Пользователь вправе запросить удаление своих данных. Нельзя просто DELETE — у нас связанные сделки, налоговые требования rs.ge (хранение 6 лет), рейтинги.

### Вопрос 1: Активные грузы при удалении

**Предложение Тимура:** автоматически отменить (`status = canceled`).

**Варианты:**
- **A (Тимур):** Автоматически отменить все `active` грузы. Плюсы: пользователь ушёл — грузы тоже. Минусы: если был надёжный перевозчик который ждёт — получит cancel без объяснений.
- **B:** Показать список активных грузов в UI до удаления, потребовать ручной отмены каждого. Плюсы: сознательное действие. Минусы: UX сложнее, можно зависнуть.

**Рекомендация:** Вариант A (автоотмена) — проще, безопаснее для базы данных.

**Решение Тимура:** _(заполнить)_

### Вопрос 2: Активные сделки при удалении

**Предложение Тимура:** заблокировать удаление, показать «Завершите сделки».

**Варианты:**
- **A (Тимур):** Блокировать удаление если есть сделки в статусе `confirmed/loading/in_transit/delivered`. Пользователь должен сначала завершить или отменить сделки. Плюсы: защита второй стороны. Минусы: пользователь может быть «заперт» если партнёр не отвечает.
- **B:** Разрешить удаление, автоматически перевести сделки в `disputed`, уведомить вторую сторону и администратора.

**Рекомендация:** Вариант A (Тимур) — надёжнее, защищает перевозчика/грузовладельца.

**Решение Тимура:** _(заполнить)_

## ADR-011: Процесс деплоя и миграций

**Дата:** 04.05.2026
**Статус:** ACCEPTED — Вариант A + D (pre-deploy Alembic + healthcheck с auto-rollback)

### Как сейчас

- Хостинг: Railway (автодеплой из GitHub `main`, Hobby $5/мес).
- Миграции: Alembic. Запускаются в `start.sh` при старте контейнера — **после** запуска нового кода.
- Порядок: старый контейнер → новый контейнер стартует → `alembic upgrade head` → FastAPI поднимается → Railway переключает трафик.

### Что произошло

При добавлении `paused` в `LoadStatus` (PostgreSQL Enum):

1. Railway задеплоил новый код — код уже знает про `paused`.
2. Контейнер стартовал, начал принимать трафик немедленно.
3. `ALTER TYPE loadstatus ADD VALUE 'paused'` ещё выполнялся — PostgreSQL не знал нового значения.
4. Все запросы к `loads` → `500` в течение ~10 секунд.

**Причина:** Railway переключает трафик до завершения `start.sh`. Нет grace period после старта.

---

### Варианты решения (с цифрами)

#### Вариант A — Pre-deploy migration step

Railway поддерживает [pre-deploy команду](https://docs.railway.app/deploy/deployments#pre-deploy-command). Миграция запускается **до** запуска нового контейнера.

```toml
# nixpacks.toml
[phases.migrate]
cmds = ["alembic upgrade head"]

[start]
cmd = "uvicorn app.main:app --host 0.0.0.0 --port $PORT"
```

| | |
|---|---|
| **Стоимость** | $0 — входит в Railway Hobby |
| **Время внедрения** | ~1 час |
| **Риск простоя** | Нет (миграция до трафика) |
| **Риск rollback** | Нужен вручную если миграция не идемпотентна |
| **Плюсы** | Решает проблему сразу, Railway нативно поддерживает |
| **Минусы** | Зависший `alembic upgrade` зависает весь деплой (нужен timeout) |

---

#### Вариант B — Backward-compatible миграции (zero-downtime)

Каждая миграция пишется так, чтобы оба поколения кода (старый и новый) работали с одной схемой. Для Enum: сначала деплоим изменение БД, потом код. Удаления колонок — два релиза.

| | |
|---|---|
| **Стоимость** | $0 |
| **Время внедрения** | 0 доп. часов (правило дисциплины) |
| **Риск простоя** | ~0 (zero-downtime по определению) |
| **Риск rollback** | Минимален — схема всегда совместима |
| **Плюсы** | Самый надёжный подход, enterprise-стандарт |
| **Минусы** | Нужна дисциплина: каждое изменение схемы = отдельный PR; Enum → VARCHAR в PostgreSQL |

---

#### Вариант C — Staging-окружение

Отдельный Railway-проект `caucashub-staging`. Каждый релиз сначала на staging, прогон smoke-тестов, потом прод.

| | |
|---|---|
| **Стоимость** | +$5–20/мес (Railway Hobby для staging + PostgreSQL $5–15) |
| **Время внедрения** | ~4 часа (создать проект, CI/CD скрипт) |
| **Риск простоя** | Снижается на 80% (проблемы видны на staging) |
| **Риск rollback** | Сохраняется, но реже нужен |
| **Плюсы** | Ловит большинство проблем до прода |
| **Минусы** | +$10–20/мес, нужна дисциплина «не пропускать staging» |

---

#### Вариант D — Healthcheck + auto-rollback

Railway проверяет `/health` после деплоя. Если 500 в течение 60 секунд — автоматический rollback на предыдущую версию.

```python
# app/main.py — уже добавить /health endpoint
@app.get("/health")
async def health(): return {"status": "ok"}
```

| | |
|---|---|
| **Стоимость** | $0 (Railway поддерживает, `/health` уже добавить ~15 мин) |
| **Время внедрения** | ~30 минут |
| **Риск простоя** | Автоматически ограничен 60 сек → rollback |
| **Риск rollback** | Rollback не отменяет уже применённую миграцию БД (!) |
| **Плюсы** | Дополнительная страховка к A или B |
| **Минусы** | Не решает race condition с миграцией, только ограничивает время простоя |

---

#### Вариант E — Комбинация B + C + D (enterprise-стандарт)

Backward-compatible миграции + staging для проверки + healthcheck с auto-rollback.

| | |
|---|---|
| **Стоимость** | +$10–20/мес (только staging + БД) |
| **Время внедрения** | ~1 день |
| **Риск простоя** | ~0 |
| **Плюсы** | Полная защита на всех уровнях |
| **Минусы** | Overhead дисциплины и стоимость staging |

---

### Рекомендация Builder

**Для текущего масштаба (до 1000 MAU) → A + D.**

- **A** решает корневую проблему (гонка миграции и трафика) — ~1 час работы, $0.
- **D** добавляет страховку — ~30 мин, $0.
- Итог: ~1.5 часа работы, $0 дополнительно.

**B подключать постепенно** по мере роста команды — как правило дисциплины, не как срочный фикс.
**C рассмотреть при 500+ MAU** когда стоимость downtime > стоимость staging.
**E — будущее состояние** при масштабировании команды.

### Принятое решение (04.05.2026)

**Тимур выбрал: Вариант A.**

**Реализовано:**
1. `nixpacks.toml` — секция `[phases.migrate]`: `alembic upgrade head` запускается до `[start]`. Если падает — деплой отменяется.
2. Бэкап: `pg_dump` в `/app/backups/pre_deploy_YYYYMMDD_HHMMSS.sql` перед каждым деплоем.
3. `GET /health` — реальная проверка соединения с БД (`SELECT 1`). Возвращает 503 если БД недоступна.
4. `app/main.py` — удалены все `ALTER TABLE / CREATE TABLE` из `lifespan`. Единственный источник правды для схемы — `alembic/versions/`.

**Ротация бэкапов:** Railway Volumes хранит файлы. Для автоматической ротации 30 дней — добавить cron-задачу или использовать Railway Volume + внешний S3 (отложено до масштабирования).

**Следующий шаг (опционально):** Вариант B как дисциплина команды — каждая деструктивная миграция = отдельный PR.

## ADR-012: Demo-режим для отображения активности до запуска

**Дата:** 04.05.2026
**Статус:** ACCEPTED

**Контекст:** До накопления реальных грузов платформа выглядит пустой. Нужны тестовые данные с явной пометкой.

**Принципы:**
1. **Явная пометка** — плашка «🟡 Демо» на карточке и в детальном виде. На грузинском: «🟡 დემო-განცხადება».
2. **Изоляция от реальных откликов** — `POST /api/responses/load/{id}` на демо-груз → 400 "Demo loads cannot receive responses".
3. **Не считается в счётчиках** — `GET /api/stats/counters` фильтрует `is_demo=False`.
4. **Не удаляется** — только помечается флагом `is_demo=True`.

**Реализация:**
- Поля: `loads.is_demo` (bool, default False), `users.is_demo` (bool, default False).
- Миграция: `005_track11_demo_mode.py` (backfill: грузы с price_gel=0 и без price_usd → is_demo=True).
- Фронт: класс `.card-demo`, бейдж `.badge-demo`, плашка `#demoBanner` в модалке.
- При клике «Откликнуться» на демо → модалка `#demoNoticeOverlay` с кнопками «Понятно» и «Регистрация».

**Срок действия:** До накопления 50+ реальных грузов в день. После — массовое удаление демо через `DELETE /api/loads/admin/bulk-delete`.

## ADR-013: Контакты грузовладельца — когда открываются перевозчику

**Дата:** 04.05.2026
**Статус:** ACCEPTED — Вариант B принят Тимуром 05.05.2026
**Решение:** Вариант B — удалить PRICING_ENABLED, контакты только через сделку.
**Реализовано:** 05.05.2026, коммит в ветке master.
**Контекст:**

### Текущее положение дел

ADR-001 (принят на старте): контакты грузовладельца доступны перевозчику только после `accepted`-отклика — то есть в рамках сделки.

ADR-009 (принят позже): добавлен флаг `PRICING_ENABLED`. При `PRICING_ENABLED=false` (текущий прод) — все авторизованные пользователи видят контакты сразу, без сделки.

Результат противоречия: биржа технически работает как «справочник контактов». Перевозчик видит телефон грузовладельца, звонит напрямую — отклика и сделки нет. Платформа теряет контроль над транзакцией, статистику, возможность будущей монетизации.

### Почему это важно решить сейчас

Первые 50 пользователей формируют паттерны поведения. Если «звонить напрямую» станет нормой, переключиться позже будет в 10 раз сложнее.

---

### Вариант A — Включить PRICING_ENABLED=true немедленно

**Суть:** Установить `PRICING_ENABLED=true` в Railway ENV. Контакты открываются только в сделке (ADR-001). Платные планы дают больше откликов в месяц, но контакты — у всех только через сделку.

**Плюсы:**
- Биржевой механизм работает с первого пользователя
- Все сделки фиксируются в системе → статистика, экспорт rs.ge
- Простота: переключить 1 переменную, ничего не кодировать

**Минусы:**
- Free plan: 0 откликов в месяц (текущий `PLAN_LIMITS["free"] = 0`) — пользователь вообще не может откликнуться без тарифа
- Нужно либо дать free plan X откликов, либо все первые пользователи зашли в «стену»
- Первые пользователи привыкли видеть контакты сразу — негативная реакция возможна

**Что сломается:** Ничего технически. Но UX ухудшится для free-пользователей пока у них 0 откликов в лимите.

**Что менять в коде:**
- Railway ENV: `PRICING_ENABLED=true`
- `PLAN_LIMITS["free"] = 5` (дать 5 бесплатных откликов в месяц, чтобы не заблокировать)
- Предупреждение в UI: «Контакты будут доступны после принятия вашего отклика»

---

### Вариант B — Удалить PRICING_ENABLED, контакты всегда только в сделке

**Суть:** Убрать флаг `PRICING_ENABLED` полностью. `show_contacts` в `GET /api/loads/{id}` всегда зависит только от наличия принятого отклика (`accepted`-отклик или активная сделка). Платные тарифы дают другие преимущества: больше откликов, приоритет в ленте, верифицированный бейдж, расширенная аналитика.

**Плюсы:**
- Архитектурно чистое решение — нет противоречия ADR-001 vs ADR-009
- ADR-009 становится неактуальным, закрывается
- Монетизация через объём откликов, а не через доступ к контактам — более понятная пользователям модель

**Минусы:**
- Требует рефакторинга `plan_check.py`, `loads.py`, всех мест где `PRICING_ENABLED`
- Нужно реализовать «контакты доступны при наличии accepted-отклика» — новая логика в API
- Free plan должен иметь ненулевой лимит откликов иначе воронка не работает

**Что ломается у существующих пользователей:**
- Никто не теряет данные. Но пользователи с текущими грузами внезапно перестанут видеть контакты — нужен grace period или backfill «все грузы у которых есть активный отклик — показывать контакты»

**Что менять в коде:**
- Удалить `PRICING_ENABLED` из `plan_check.py` и `loads.py`
- Добавить новую логику: `show_contacts = user has accepted response to this load OR user is in deal`
- Обновить `GET /api/loads/{id}` — JOIN с responses
- Обновить UI: убрать условия PRICING_ENABLED в `main.js`
- Оценка: ~4 часа

---

### Вариант C — Сохранить текущее (контакты открыты сразу)

**Суть:** Принять что биржа на MVP работает как «доска объявлений с контактами». ADR-001 отменяется. Монетизация идёт через другие механизмы: платное поднятие объявления, реклама, страхование грузов, верификация.

**Плюсы:**
- Нулевой барьер для пользователей — максимальная конверсия при первом запуске
- Никаких изменений в коде
- Перевозчики доверяют бирже больше, если видят контакты сразу

**Минусы:**
- Нет механизма принудительного использования откликов → нет данных о сделках → нет экспорта rs.ge с реальными данными
- Монетизация через «доску объявлений» — намного сложнее, чем через биржевой механизм
- Нельзя будет разрабатывать рейтинги, статистику, защиту от мошенников
- ADR-001, ADR-009, значительная часть категорий 4–6 написаны под биржевую модель — они теряют смысл

**Что ломается:** Ничего технически. Но стратегически — потеря биржевой модели.

**Что менять в коде:**
- Удалить `PRICING_ENABLED`, `is_paid_plan`, `check_can_respond` — упростить
- Убрать paywall UI
- Оценка: ~2 часа (упрощение)

---

### Рекомендация Builder

**Рекомендую Вариант A с небольшим дополнением.**

Включить `PRICING_ENABLED=true` + дать free plan 5–10 откликов/месяц. Это защищает биржевой механизм с первого дня, не создаёт непреодолимого барьера для новых пользователей и не требует рефакторинга.

Вариант B — архитектурно правильнее, но требует 4+ часов работы и может создать регрессию.

Вариант C — потеря сути продукта.

### Вопрос для Тимура

- **A** — `PRICING_ENABLED=true` + free plan 5–10 откликов (рекомендую, готово за 30 мин)
- **B** — Удалить PRICING_ENABLED, контакты только через сделку (~4 часа кода)
- **C** — Оставить как есть, принять что это «доска объявлений»

## ADR-016: Архитектура двусторонней биржи

**Дата:** 05.05.2026
**Статус:** PROPOSED — ожидает подтверждения Тимура
**Автор:** Builder (ассист) по решениям Тимура

### Контекст

CaucasHub запущена как односторонняя биржа: грузовладелец размещает груз, перевозчик откликается.
Раздел «Транспорт» существует но используется 0/30 пользователей.

**Решение Тимура (05.05.2026):** Развить в симметричную двустороннюю биржу:
- Перевозчик размещает **транспортное предложение** (маршрут, дата, тип, вместимость)
- Грузовладелец видит предложения и **откликается** на них
- Создаётся **сделка** — такая же как при откликах на груз

---

### ADR-016.1 — Единая модель Deal с двумя источниками

**Решение:** Расширить таблицу `deals` двумя nullable FK:

```sql
ALTER TABLE deals ADD COLUMN cargo_id INTEGER REFERENCES loads(id);
ALTER TABLE deals ADD COLUMN transport_offer_id INTEGER REFERENCES transport_offers(id);
ALTER TABLE deals ADD COLUMN response_id INTEGER REFERENCES responses(id);
ALTER TABLE deals ADD COLUMN transport_request_id INTEGER REFERENCES transport_requests(id);
-- CHECK: ровно один источник заполнен
```

**Инварианты:**
- `(cargo_id IS NOT NULL) XOR (transport_offer_id IS NOT NULL)` — сделка либо из груза, либо из транспорта
- `shipper_id`, `carrier_id`, статусная машина, `final_price_*` — общие для обоих типов

**Обратная совместимость:** Существующие сделки → `cargo_id` заполнен, `transport_offer_id = NULL`.

---

### ADR-016.2 — Раздельные модели отклика

**Решение:**

| Модель | Кто создаёт | На что | FK |
|--------|-------------|--------|----|
| `Response` (существующая) | Перевозчик | Груз (`Load`) | `load_id` → `loads.id` |
| `TransportRequest` (новая) | Грузовладелец | Транспортное предложение (`TransportOffer`) | `transport_offer_id` → `transport_offers.id` |

`TransportRequest` поля: `id, user_id (shipper), transport_offer_id, price_usd, message, status (pending/accepted/rejected/cancelled), created_at`

Не объединять в одну таблицу: разные участники, разная семантика.

---

### ADR-016.3 — Раздельные подписки на уведомления

**Решение:**

| Модель | Кто подписывается | Хук | Когда срабатывает |
|--------|-------------------|-----|--------------------|
| `RouteSubscription` (существующая) | Перевозчик | `POST /api/loads/` | Новый груз по маршруту |
| `TransportSubscription` (новая) | Грузовладелец | `POST /api/transport/` | Новое транспортное предложение по маршруту |

`TransportSubscription` поля идентичны `RouteSubscription`: `user_id, from_city, to_city, notify_tg, notify_email, truck_type, max_weight_t, is_active`.

---

### ADR-016.4 — Мари определяет тип объявления по роли

**Решение:**

```
Пользователь: «Везу из Тбилиси в Батуми тент, 5 тонн, 15 мая»
Мари (если роль = carrier): → строит TransportOffer, предлагает «Разместить транспорт?»
Мари (если роль = shipper):  → строит Load, предлагает «Разместить груз?»
Пользователь может явно написать «хочу груз» / «хочу машину» — переопределяет.
```

Реализация: в `ai.py` при `aiPostLoad()` проверяем `user.role`, выбираем тип формы.

---

### ADR-016.5 — Симметрия в UI

**Решение:**

| Компонент | Грузы (существует) | Транспорт (новый) |
|-----------|-------------------|-------------------|
| Публичный список | `/` вкладка «Грузы» | `/` вкладка «Транспорт» |
| Карточка | `renderLoadCard()` | `renderTruckOfferCard()` |
| Форма размещения | `openPostLoad()` | `openPostTransport()` |
| Фильтры | from/to/type/weight | from/to/type/weight |
| Кабинет | вкладка «📦 Грузы» | вкладка «🚛 Транспорт» |
| Отклики в кабинете | вкладка «Отклики» (перевозчика) | вкладка «Запросы» (грузовладельца на транспорт) |

Нижняя навигация: «🚛 Машины» остаётся, но теперь ведёт на публичную биржу транспорта.

---

### ADR-016.6 — Контакты по сделке (наследуется из ADR-013 B)

**Решение:** Идентично ADR-013:

- `GET /api/transport/{id}` → `owner_phone = None` всегда
- `GET /api/deals/my` → контакты видны если `viewer = shipper_id OR carrier_id`
- Отклик `TransportRequest` создаётся свободно (нет платного барьера)

**Уточнение: роль `both`**
Пользователь с ролью `both` (одновременно перевозчик и грузовладелец) может быть
на любой стороне сделки — в зависимости от того, кто инициировал.
В `Deal` всегда явно фиксируется `shipper_id` и `carrier_id`.
**CHECK constraint:** `shipper_id != carrier_id` — один user не может быть обеими сторонами одной сделки.

---

### ADR-016.7 — Экспорт rs.ge для двусторонних сделок

**Решение:** Экспорт включает все `Deal` независимо от источника (`cargo` или `transport_offer`).

Поля экспорта общие:
```
deal_id, act_number, created_at, completed_at
shipper.tax_id, shipper.legal_name
carrier.tax_id, carrier.legal_name
route: from_city → to_city  (из cargo если есть, иначе из transport_offer)
final_price_gel, final_price_usd, exchange_rate_snapshot
```

**Реализация:** В `GET /api/deals/export` — добавить JOIN с `transport_offers`
для заполнения from_city/to_city когда `cargo_id IS NULL`.

**Тестовый сценарий:**
1. Создать сделку через TransportRequest-флоу (transport_offer_id заполнен)
2. Завершить сделку (status = completed)
3. `GET /api/deals/export` → запись присутствует с правильными полями маршрута
