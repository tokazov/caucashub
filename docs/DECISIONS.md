## ADR-006: Валюта на платформе

**Дата:** 03.05.2026  
**Статус:** PROPOSED — требуется решение Тимура  
**Контекст:** В модели `Load` два поля — `price_usd` и `price_gel`. В модели `Response` только `price_usd`. При принятии отклика сделка может получить цену в разных валютах — груз в ₾, отклик в $. В `Deal.currency` поле есть, но `agreed_price` берётся из отклика (всегда USD) или груза (может быть GEL). Фронт отображает `₾` или `$` в зависимости от того, какое поле заполнено.

**Проблема:** В одной сделке возможен микс валют. Экспорт для rs.ge некорректен, если стороны согласовали цену в разных валютах.

---

### Вариант A — Всё в лари (GEL), USD только для справки

**Суть:** `Load.price_gel` — основное поле, `price_usd` и `Response.price_usd` переименовать в `price_gel`. Хранить только GEL. Таблицы ставок ($/км) — только для отображения, не хранить в БД.

**Плюсы:**
- Грузия — страна с лари. Для местных сделок GEL логичнее.
- Простая логика: одна валюта, нет конвертации.
- rs.ge экспорт → всегда GEL, без вопросов.
- Минимум изменений в UI.

**Минусы:**
- Международные маршруты (Россия, Турция) — перевозчики привыкли считать в USD/RUB.
- Нет возможности зафиксировать долларовую договорённость.
- При инфляции лари — цены надо пересчитывать вручную.

**Что мигрировать:**
- `Response.price_usd` (Integer!) → `price_gel` (Float)
- Все записи в таблице `responses` — конвертировать по курсу на дату создания (или принять as-is, если немного данных)
- Убрать `Load.price_usd` из API-ответов или оставить для отображения
- Фронт: убрать `cur: "$"` ветку, всегда показывать `₾`

**Время реализации:** ~2-3 часа

---

### Вариант B — Всё в USD, лари только отображение

**Суть:** Единственная база-валюта — USD. На фронте конвертировать в GEL по курсу (из внешнего API или захардкоженному).

**Плюсы:**
- Стабильнее для межнациональных маршрутов.
- Перевозчики из РФ/Турции привыкли к USD.

**Минусы:**
- rs.ge требует GEL для отчётности. Нужна конвертация в момент экспорта.
- Нужен курс USD/GEL (Нацбанк Грузии API или захардкодить — оба варианта плохие).
- Юридически договоры в Грузии лучше в GEL.
- Клиенты-грузины видят цену в $ — непривычно.

**Что мигрировать:**
- `Load.price_gel` → конвертировать в USD и сохранить в `price_usd`
- Добавить `exchange_rate` snapshot в `Deal` (курс на момент сделки)
- Подключить API курсов (NBG API бесплатный: nbg.gov.ge)

**Время реализации:** ~1 день + интеграция с API курсов

---

### Вариант C — Две валюты + снапшот курса в момент сделки (рекомендован для будущего)

**Суть:** Хранить цену в валюте которую выбрал пользователь + добавить в `Deal` поля `price_gel_snapshot` и `exchange_rate_snapshot`. При создании сделки — записать курс на этот момент.

**Плюсы:**
- Максимальная гибкость: договорились в $, зафиксировали в ₾ для rs.ge.
- Исторически точно: курс зафиксирован на дату.
- Никто ничего не теряет при колебании курса.

**Минусы:**
- Сложнее реализация.
- Нужен курс NBG в реальном времени.
- Два поля цены везде — UX усложняется.

**Что мигрировать:**
- Добавить в `Deal`: `price_gel_snapshot Float`, `price_usd_snapshot Float`, `exchange_rate Float`
- Добавить в `Response`: поле `currency String` (GEL/USD)
- При `accept_response` → запросить NBG API и записать снапшот

**Время реализации:** ~1-2 дня

---

### Рекомендация (не решение)

Для **быстрого запуска** — Вариант A (всё в GEL): минимум изменений, Грузия = GEL.
Для **масштабирования** — Вариант C через 3-6 месяцев после первых международных сделок.

**Решение принимает:** Тимур  
**Дедлайн решения:** до старта рекламных кампаний

## ADR-007: Справочник городов

**Дата:** 03.05.2026  
**Статус:** PROPOSED — требуется решение Тимура  
**Контекст:** `Load.from_city`, `Load.to_city`, `Truck.available_from`, `Truck.available_to` — все хранятся как свободный текст. В БД уже могут быть записи «Тбилиси», «тбилиси», «Тбилилсси», «Tbilisi». Фильтр `ilike` частично спасает, но: агрегация по городам сломана, автокомплит нет, опечатки попадают в БД.

**Проблема:** Нет нормализации → фильтры неточные, статистика врёт, поиск «из Тбилиси» не находит «tbilisi».

---

### Вариант A — Захардкодить топ-50 городов (JSON-справочник)

**Суть:** Создать файл `app/data/cities.json` с топ-50 городами (Грузия, Россия/СНГ топ-20, Турция топ-5). На фронте — datalist или select. API валидирует против этого списка.

```json
[
  {"id": "tbilisi", "label_ru": "Тбилиси", "label_ge": "თბილისი", "country": "GE"},
  {"id": "batumi",  "label_ru": "Батуми",  "label_ge": "ბათუმი",  "country": "GE"},
  ...
]
```

**Плюсы:**
- Реализация за 2-3 часа.
- Нет внешних зависимостей и расходов.
- Фильтры работают точно.
- Покрывает 95%+ реальных маршрутов.

**Минусы:**
- Жёсткий список: нельзя добавить новый город без деплоя.
- Для нетипичных маршрутов — пользователь застрял.
- Нужно переиндексировать старые записи в БД (разовая задача).

**Стоимость:** $0. Время: 3-4 часа (JSON + валидация на бэке + datalist на фронте).

---

### Вариант B — Google Places API с автокомплитом

**Суть:** Поле города → текст с автокомплитом через Google Places API. При выборе — сохранять `place_id` + нормализованное название.

**Плюсы:**
- Любой город мира.
- UX: привычный автокомплит как в Google Maps.
- Нормализация гарантирована Google.

**Минусы:**
- **Стоимость:** $0.003/запрос. При 1000 пользователях и 5 запросах каждый — $15/мес минимум, растёт с трафиком.
- Нужен API ключ, биллинг аккаунт Google Cloud.
- При превышении лимита — форма ломается.
- GDPR: запросы идут через Google.
- Зависимость от внешнего сервиса.

**Стоимость:** $15-100/мес + разработка 1 день.

---

### Вариант C — Таблица в БД с админкой

**Суть:** Таблица `cities(id, name_ru, name_ge, country, is_active)`. Админ добавляет города через панель. Бэк валидирует поле города против этой таблицы.

**Плюсы:**
- Гибко: можно добавлять города без деплоя.
- Нет внешних зависимостей.
- Статистика по городам точная.

**Минусы:**
- Нужна админка (ещё не реализована).
- Время реализации: 1-2 дня (таблица + CRUD + валидация + миграция).
- До заполнения таблицы — фича не работает.

**Стоимость:** $0 + 1-2 дня разработки.

---

### Вариант D — Свободный текст, но нормализация (lowercase + trim)

**Суть:** Оставить свободный ввод. При сохранении: `city = city.strip().title()` (первая буква заглавная, остальные маленькие). Добавить `ilike` в фильтрах (уже есть).

**Плюсы:**
- Нет изменений в UX.
- Реализация за 30 минут.
- Покрывает 80% дублей (опечатки — нет, но капитализация — да).

**Минусы:**
- «Тбилиси» и «Tbilisi» — всё равно разные записи.
- Опечатки в БД остаются.
- Нет автокомплита — пользователь вводит руками.
- Агрегация по городам по-прежнему неточная.

**Стоимость:** $0. Время: 30 минут.

---

### Рекомендация (не решение)

**Быстрый старт:** Вариант D (30 мин) + Вариант A параллельно (3-4 часа).  
**Стратегически:** Вариант A — оптимален для текущего масштаба (Грузия + соседи).  
Вариант B — только если трафик вырастет до 10K+ MAU и есть рекламный бюджет.

**Решение принимает:** Тимур  
**Зависимость:** ADR-006 (валюта) не зависит от этого решения, можно решать параллельно.
